# Sevana Database Design Document

Version: 1.0

Status: Approved

Owner: Sevana Engineering Team

---

# 1. Purpose

This document defines the database architecture of Sevana.

It serves as the single source of truth for:

- Database schema
- Relationships
- Constraints
- Naming conventions
- Scalability decisions
- Security considerations

All backend development must follow this document.

---

# 2. Design Philosophy

The database is the heart of Sevana.

Every rescue, notification, participant, timeline event and user action should be represented as structured data.

The database must prioritize:

- Data integrity
- Performance
- Auditability
- Scalability
- Security

---

---

# 5. Timestamp Strategy

Every major table should contain

```
created_at

updated_at
```

Whenever applicable

```
deleted_at
```

should be used instead of permanent deletion.

---

# 6. Soft Delete Policy

Critical rescue data should never be permanently deleted.

Instead

```
deleted_at TIMESTAMPTZ
```

will be added where necessary.

Reasons

- audit
- analytics
- recovery
- legal records
- abuse investigation

# 30. Security Rules

Never trust frontend data.

Validate

- IDs
- ownership
- permissions

Use foreign keys.

Never expose password hashes.

Never store sensitive information in plaintext.

---

# 31. Database Principles

1. Database is the source of truth.

2. Never overwrite historical rescue information.

3. Every important action must be traceable.

4. Every relationship should be normalized.

5. Optimize reads without sacrificing integrity.

6. Build for scale from the beginning.
